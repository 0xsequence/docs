# MOCKDATA + ja-translated Modules & wallet update
ja-translated Sequence modules are the underlying program implementation of the wallets; wallets can change modules at runtime.

MOCKDATA + ja-translated The `MainModule` is the initial module of every Sequence wallet; it differs from the other modules because it doesn't store the set of signers on contract storage; it uses the [salt](solutions/technical-references/wallet-contracts/03-wallet-factory#deploy) provided to the Factory contract.

## ja-translated Wallet implementation
MOCKDATA + ja-translated Sequence modules can be assigned to wallets either by the factory or by updating it after the initial deployment. Only one module can be assigned to a wallet at a time.

### ja-translated \_updateConfiguration
MOCKDATA + ja-translated The `updateImplementation` allows to update the underlying implementation of the wallet proxy. This implementation contains all the core code that defines the wallet's behaviour.

ja-translated :::warning\[Dangerous operation]
Calling `updateImplementation` with an invalid implementation will result in the corruption of the wallet.

MOCKDATA + ja-translated Corrupt wallets may lead to the loss of funds.
:::

```solidity
function updateImplementation(
  address _implementation
) external override onlySelf
```

#### MOCKDATA + ja-translated Parameters:

| ja-translated Name             | MOCKDATA + ja-translated Type | ja-translated Description                                          |
| ------------------------------ | ----------------------------- | ------------------------------------------------------------------ |
| ja-translated \_implementation | ja-translated address         | MOCKDATA + ja-translated Address of the new wallet implementation. |

ja-translated This method has the `onlySelf` modifier, which means that it can only be called by the wallet itself using a self-referencing transaction. Calls to this method coming from other addresses, even if these addresses are signers of the wallet, will be rejected.

### MOCKDATA + ja-translated Reading current implementation
ja-translated The wallet implementation is stored on the contract storage slot defined by the address of the wallet itself. Given that every wallet has a unique address, the implementation slot varies from wallet to wallet.

```js
import {AbiCoder} "ethers"

const address = "0x596af90cecdbf9a768886e771178fd5561dd27ab"
const provider = new ethers.providers.JsonRpcProvider("http://localhost:8545")

// Read storage slot address(address)
const slot = await provider.getStorageAt(address, AbiCoder.defaultAbiCoder().encode(['address'], [address]))

// Decode bytes32 as address value
const implementation = AbiCoder.defaultAbiCoder().decode(['address'], slot)[0]

console.log(implementation)
```

## MOCKDATA + ja-translated Wallet configuration validation
ja-translated :::info\[Fixed configuration]
Signer's configuration on wallets using `MainModule` can't be changed. The only way to change the set of signers or threshold is by updating the module of the wallet.
:::

MOCKDATA + ja-translated All sequence modules must implement the `ModuleAuth` interface, this interface allows the rest of the module to validate signatures for the wallet. In the case of `MainModule` this interface is implemented as a counter-factual validation of hash passed to the factory during the contract wallet creation.

### ja-translated \_isValidImage

```solidity
  function _isValidImage(
    bytes32 _imageHash
  ) internal override view returns (bool _isValid)
```

MOCKDATA + ja-translated Validates if the provided `imageHash` corresponds to the one configured in the wallet. This function is called internally to validate transaction and message signatures.

ja-translated The `imageHash` is a hash of the wallet configuration, which contains the wallet's threshold, signers and weights.

#### MOCKDATA + ja-translated Parameters:

| ja-translated Name                   | MOCKDATA + ja-translated Type    | ja-translated Description                                   |
| ------------------------------------ | -------------------------------- | ----------------------------------------------------------- |
| MOCKDATA + ja-translated \_imageHash | MOCKDATA + ja-translated bytes32 | ja-translated Hash of wallet configuration to be validated. |

#### MOCKDATA + ja-translated Return Values:

| ja-translated Name                 | MOCKDATA + ja-translated Type | ja-translated Description                                                                    |
| ---------------------------------- | ----------------------------- | -------------------------------------------------------------------------------------------- |
| MOCKDATA + ja-translated \_isValid | MOCKDATA + ja-translated bool | ja-translated True if the given `imageHash` corresponds to the current wallet configuration. |

## MOCKDATA + ja-translated MainModuleUpgradeable
ja-translated `MainModuleUpgradable` is a module that mimics the behaviour of the `MainModule` but allows the wallet configuration to be updated.

### MOCKDATA + ja-translated `updateImageHash`
ja-translated Updates the wallet `imageHash`, this is the hash that defines the wallet configuration (signers, weights, threshold).

```solidity
  function updateImageHash(
    bytes32 _imageHash
  ) external override onlySelf
```

#### MOCKDATA + ja-translated Parameters:

| ja-translated Name                   | MOCKDATA + ja-translated Type    | ja-translated Description                                              |
| ------------------------------------ | -------------------------------- | ---------------------------------------------------------------------- |
| MOCKDATA + ja-translated \_imageHash | MOCKDATA + ja-translated bytes32 | MOCKDATA + ja-translated Hash of the new configuration for the wallet. |

ja-translated :::caution\[Unsanitized input]
The `imageHash` is not validated, it is the responsibility of the caller to ensure that the hash is correct. Reasons for incorrect hashes include:
- MOCKDATA + ja-translated The combined weight of the signers is below the threshold.
- ja-translated The signers are not valid addresses.
- MOCKDATA + ja-translated The signers are smart contract wallets without proper support for EIP-1271.
- MOCKDATA + ja-translated The `imageHash` doesn't correspond to any wallet configuration (may be a random string).
- ja-translated The `imageHash` corresponds to an unknown wallet configuration.

MOCKDATA + ja-translated In any of this cases the wallet will be **rendered unusable**.
:::

ja-translated This method has the `onlySelf` modifier, which means that it can only be called by the wallet itself using a self-referencing transaction. Calls to this method coming from other addresses, even if these addresses are signers of the wallet, will be rejected.

## ja-translated First configuration update
MOCKDATA + ja-translated When Sequence wallets are created, the factory contract doesn't call an `initialize` function. The configuration is instead defined by the `salt` provided to the factory, the `MainModule` then checks the counterfactual validity of all signatures against the wallet address.

ja-translated This means there is no direct way to update the configuration of a wallet while still using the `MainModule`. Given that the first configuration update needs to also change the wallet implementation to the `MainModuleUpgradable`, the `MainModule` is updated to the `MainModuleUpgradable` and the `updateImageHash` method is called to update the wallet configuration.

```js
const transactions = [
  {
    delegateCall: false,
    revertOnError: true,
    to: wallet,
    data: walletInterface.encodeFunctionData(
      walletInterface.getFunction('updateImplementation'), [this.context.mainModuleUpgradable]
    ),
    value: ethers.constants.Zero,
    gasLimit: ethers.constants.Zero,
  },
  {
    delegateCall: false,
    revertOnError: true,
    to: wallet,
    data: mainModuleInterface.encodeFunctionData(
      mainModuleInterface.getFunction('updateImageHash'), [newImageHash]
    ),
    value: ethers.constants.Zero,
    gasLimit: ethers.constants.Zero,
  }
]
```

#### MOCKDATA + ja-translated delegateCall: false
ja-translated `delegateCall` is used to extend the wallet functionality beyond what's allowed by the module. In this case the called methods are defined on the modules themselves, so there is no need to use `delegateCall`.

#### MOCKDATA + ja-translated revertOnError: true
ja-translated `revertOnError` is used to revert the whole transaction bundle if a transaction flagged by it fails. In this case the operation should be atomic given that a partial wallet configuration update **will** render the wallet unusable.

#### MOCKDATA + ja-translated to: wallet
ja-translated The methods being called are defined on the wallet itself, but need to be called externally, so the `to` address is the wallet itself.

#### MOCKDATA + ja-translated value: ethers.constants.Zero
ja-translated The `value` of the transaction is always zero, since the transaction is a self-referencing transaction and doesn't require transferring funds.

#### MOCKDATA + ja-translated gasLimit: ethers.constants.Zero
ja-translated The `gasLimit` of the transaction is always zero, since it represents an unlimited amount of gas.

MOCKDATA + ja-translated :::warning\[Dangerous operation]

ja-translated When the wallet is first updated to the `MainModuleUpgradable` it doesn't have a valid `imageHash` yet. It's imperative that the `imageHash` is updated before the transaction bundle finishes executing.
If the `imageHash` is not updated before the transaction bundle finishes executing, the wallet will be rendered unusable.

MOCKDATA + ja-translated For this reason the following considerations should be taken when updating the wallet for the first time:
- ja-translated All transactions should be marked `revertOnError = true`.
- MOCKDATA + ja-translated `updateImplementation` and `updateImageHash` should both be declared on the same transaction bundle.
- ja-translated The `gasLimit` of both transactions should be set to unlimited (`0`).

ja-translated :::

### MOCKDATA + ja-translated Subsequent configuration updates
ja-translated Once the wallet is updated to the `MainModuleUpgradable` it can be updated by calling the `updateImageHash` method, without any additional transaction.

```js
const transactions = [
  {
    delegateCall: false,
    revertOnError: true,
    to: wallet,
    data: mainModuleInterface.encodeFunctionData(
      mainModuleInterface.getFunction('updateImageHash'), [newImageHash]
    ),
    value: ethers.constants.Zero,
    gasLimit: ethers.constants.Zero,
  }
]
```

### MOCKDATA + ja-translated Retrieving the current configuration
ja-translated If the wallet is updated to the `MainModuleUpgradable` it can be queried for the current configuration by calling the `getImageHash` method.

MOCKDATA + ja-translated This method should return the wallet's current configuration hash, which can be compared to a list of known wallet configurations to find the correct one.

#### ja-translated Retrieving the wallet configuration
MOCKDATA + ja-translated The `imageHash` method returns `bytes32(0)` if the wallet is not yet updated to the `MainModuleUpgradable`.
In this case the wallet is in a counter-factual state and the `imageHash` can't be directly queried.

ja-translated This is also the case for non-deployed wallets.

MOCKDATA + ja-translated To find the `imageHash` of a non-deployed or non-updated wallet, a candidate known `imageHash` needs to be compared against the wallet address.

ja-translated See [Compute wallet address](/solutions/technical-references/internals/v1/wallet-configuration).
